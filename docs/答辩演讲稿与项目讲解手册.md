## 0. 先统一口径

### 0.1 一句话说明项目

这个项目模拟的是即刻、小红书一类内容社区里的“实时热点榜单引擎”。用户对话题进行点赞、收藏、转发、评论后，系统会把这些互动事件保存下来，再按互动权重、发布时间衰减、防刷规则计算热度分，最后生成全站榜、圈子榜、新人榜、飙升榜和个性化榜单，并通过前端页面展示出来。

上台时不要只说“这是一个 Spring Boot 项目”。更准确的说法是：

> 我们做的是一个内容社区实时热榜系统。它不是简单 CRUD，而是围绕“用户互动如何影响榜单排名”做了一整套链路，包括互动写入、反刷校验、热度聚合、缓存加速、多维榜单、前端展示和测试验证。

### 0.2 答辩主线

整场答辩建议按这条线讲：

1. 先讲业务问题：为什么社区需要热榜，为什么不能简单按点赞数排。
2. 再讲系统链路：用户互动怎样从前端进入后端，怎样影响数据库、缓存和榜单。
3. 再讲后端核心：热度算法、多维榜单、反刷、缓存、Redis、个性化。
4. 再讲前端：页面如何通过 API 展示榜单，互动模拟如何触发后端。
5. 再讲测试：Apifox 接口测试、互动模拟、JMeter 压测。
6. 最后讲交付：中文文档、演示脚本、分工、可扩展方向。

## 1. 项目小白速成

### 1.1 热榜到底是什么

热榜就是把社区里“当前最值得关注的内容”排出来。比如一个帖子刚发布，如果很多人立刻评论和转发，它应该上升得很快；另一个帖子虽然历史点赞很多，但已经过了很久，如果现在没人互动，就不应该一直霸榜。

所以热榜不是历史总榜。历史总榜看“累计有多少互动”，实时热榜看“现在是不是正在变热”。

### 1.2 为什么不能只看点赞数

只看点赞数有三个问题：

第一，点赞成本低。用户顺手点一下赞，不一定代表内容真的很有讨论价值。

第二，旧内容容易长期霸榜。一个很早发布的话题，如果历史点赞数很多，即使现在没人看，仍然可能排在前面。

第三，容易被刷。机器账号批量点赞会把一个话题顶上去。

所以项目里使用加权互动和时间衰减：

```text
有效互动分 = 点赞 * 1 + 收藏 * 2 + 转发 * 3 + 评论 * 5
热度分 = 有效互动分 / (发布时间距离当前小时数 + 2)^1.8
```

如果有可疑行为，还会乘上 `weight_multiplier` 做降权。

### 1.3 MySQL 和 Redis 的分工

MySQL 是事实库。它保存真实数据，例如话题、互动事件、用户行为、榜单快照。答辩时可以说：MySQL 负责“可追溯、可审计、可修复”。

Redis 是高速排名结构。它适合保存排行榜这种高频读取数据，项目里用 Redis ZSet 作为榜单对比和缓存通道。答辩时可以说：Redis 负责“读得快、排名方便”。

为什么两个都要用？因为热点榜单既要快，也要可信。Redis 快，但不能只靠 Redis 保存全部事实；MySQL 稳，但每次都查 MySQL 排序会增加数据库压力。

### 1.4 什么是反刷

反刷就是防止用户或机器通过异常互动刷高热度。7 天文档提到三类典型刷量：

1. 同一用户短时间内对同一话题反复互动。
2. 批量小号或相同设备集中互动。
3. 某个话题一小时内互动量异常突增。

项目里的处理方式不是一刀切。强风险，比如同用户频率超限，可以直接拒绝进入有效互动流水；弱风险，比如同设备多账号，可能是网吧、实验室、家庭共享设备，所以更适合降权，不直接拒绝。

### 1.5 什么是缓存不一致窗口

互动写入后，不一定立刻改变榜单排名。因为项目是“事件先入库，聚合任务再计算热度”。也就是说：

```text
互动写入 interaction_event
      ↓
定时或手动触发聚合
      ↓
更新 topic.current_score
      ↓
刷新榜单缓存和 Redis
      ↓
前端看到排名变化
```

这中间存在短暂不一致窗口。课程项目里这是可以接受的，因为目标是“分钟级热度更新”，不是银行转账那种强一致。

如果老师追问，可以答：

> 当前项目通过秒级本地缓存 TTL、聚合后主动清理缓存、Redis 同步来缩短不一致窗口。业务上热点榜单允许近实时，不要求每一次互动立刻同步到所有榜单。真正高并发场景下，可以用消息队列和 Redis 主读链路进一步缩短延迟。

## 2. 逐页演讲稿

下面按优化版 PPT 的 45 页来写。每页包括“讲稿”“小白解释”“可能追问”。正式上台时不一定逐字念，但建议先按这个稿子通读两遍。

## 第 1 页：项目标题

### 这一页想表达什么

告诉老师：我们做的是一个完整的内容社区实时热榜系统，核心不是页面好看，而是从用户互动到榜单变化的完整链路。

### 推荐讲稿

各位老师好，我们组答辩的项目是“即刻 App 内容社区实时热点榜单引擎”。这个项目模拟的是兴趣社区里的热榜系统，比如用户对一个话题点赞、收藏、转发、评论后，这些行为会影响话题热度，系统需要在较短时间内把热度变化反映到榜单上。

我们这次展示会围绕一条完整链路展开：项目启动后，用户在前端页面产生互动；后端接收互动并做反刷校验；有效互动进入数据库；热度聚合任务计算分数；榜单刷新到 MySQL 和 Redis；最后前端页面展示全站榜、圈子榜、飙升榜等结果。

### 小白解释

这一页就是“先给全场定调”。不要讲太多技术细节，先让老师知道这是一个热榜引擎，不是普通信息管理系统。

### 可能追问

问：你们这个项目和普通 CRUD 有什么区别？  
答：普通 CRUD 主要是增删改查。本项目除了保存数据，还要根据、处理时间互动事件计算热度衰减、识别刷量、维护缓存、生成多种榜单，并且要用测试和压测验证性能。

## 第 2 页：系统解决的问题

### 推荐讲稿

这一页主要说明项目的业务背景。社区热榜有三个核心问题。第一，互动不能简单等价，因为点赞、收藏、转发、评论的用户成本不同。第二，热榜要反映“正在热”，不能让历史高互动内容长期霸榜。第三，榜单需要抵抗刷量，否则机器人刷赞会污染排序。

所以我们的核心目标是：把社区里的每一次互动，在分钟级内转成可信、可解释、可查询、可展示的榜单结果。

### 小白解释

“可信”指不能被随便刷。  
“可解释”指能说清楚为什么这个话题排第 1。  
“可查询”指前端和接口能查到结果。  
“可展示”指最终能在网页上让老师看到榜单变化。

### 可能追问

问：为什么评论权重最高？  
答：评论需要用户输入内容，参与成本最高，也更能代表讨论热度，所以权重设为 5。点赞成本最低，权重为 1。收藏和转发介于中间，分别是 2 和 3。

## 第 3 页：答辩讲述顺序

### 推荐讲稿

为了让大家更容易理解，我们不按代码目录讲，而是按系统运行过程讲。第一部分由架构同学讲项目从启动到网站互动的完整链路。第二部分由后端同学讲 7 天任务中的核心业务逻辑，包括算法、多榜单、反刷、缓存、Redis。第三部分由前端同学讲前后端对接和页面展示。第四部分讲 Apifox 接口测试。第五部分讲互动模拟和 JMeter 压测。最后讲交付文档、分工和总结。

### 小白解释

老师听答辩时最怕听到“这个类调用那个类”的流水账。我们按“用户做了什么，系统发生什么，最后看到什么结果”讲，更容易被理解。

### 可能追问

问：为什么架构先讲？  
答：因为组员分工很多，如果先讲局部实现，听众可能不知道这些功能解决什么问题。先讲整体链路，可以让后面的算法、接口和测试都有上下文。

## 第 4 页：架构部分开场

### 推荐讲稿

接下来先进入架构部分。我们会先用一条完整链路说明项目怎么工作。项目不是一个单独接口，而是前端、后端、缓存、数据库共同协作。理解这条链路后，后面的热度算法、反刷和压测就更容易理解。

### 小白解释

这一页相当于换章节。讲的人可以停顿一下，告诉老师“下面先看系统怎么跑起来”。

### 可能追问

问：你们的链路里最核心的数据是什么？  
答：最核心的是互动事件，也就是 `interaction_event`。每次点赞、收藏、转发、评论都会形成事件，后续热度计算都基于这些事件。

## 第 5 页：四层协作

### 推荐讲稿

项目启动后主要形成四层。前端层是 Vue Dashboard，负责展示榜单和模拟互动。接口层是 Spring MVC Controller，接收前端和测试工具的请求。服务层包括 Ranking、Interaction、AntiSpam 等业务服务，负责热度计算、反刷、榜单查询和偏好更新。存储层包括 MySQL 和 Redis，MySQL 保存事实数据，Redis 保存高频读取的排名数据。

这四层的关系可以概括为：前端发请求，后端做业务，MySQL 存事实，Redis 加速读。

### 小白解释

可以把系统想成餐厅：前端是点餐界面，Controller 是服务员，Service 是后厨，MySQL 是账本，Redis 是常用菜单缓存。

### 可能追问

问：Redis 为什么不是必须的数据库？  
答：Redis 主要用于高频读取和排名加速。真正的互动流水、话题状态、快照记录仍然保存在 MySQL，保证数据可审计和可恢复。

## 第 6 页：互动写入链路

### 推荐讲稿

这一页讲一次用户互动如何进入系统。用户在页面点击互动按钮，比如评论一个话题，前端会调用 `POST /api/interaction`。后端先检查参数和话题状态，再进入反刷规则判断。如果行为有效，就写入互动事件表；同时记录用户行为审计，更新用户圈子偏好，并让榜单缓存失效。

这里要注意，写入互动不等于马上改榜单排名。它先保存事实，后面由聚合任务统一计算热度。

### 小白解释

为什么不每次互动都直接改分？因为高并发下每次都立即重算榜单会很重，而且不好追溯。先保存事件，再批量聚合，是热榜系统里常见做法。

### 可能追问

问：无效互动会不会完全丢掉？  
答：不会。频率超限这类强风险不会进入有效互动流水，但会写入 `user_behavior` 审计表，保留无效原因，方便生成反刷报告。

## 第 7 页：聚合和查询链路

### 推荐讲稿

互动事件写入后，榜单变化发生在聚合链路和查询链路之间。聚合任务会读取一定时间范围内的互动事件，按不同互动权重和反刷倍数计算有效互动分，再结合发布时间做时间衰减，更新 `topic.current_score`。同时系统会生成榜单快照，并把正常话题同步到 Redis。

查询链路则负责把榜单结果返回给前端。高频榜单查询会优先使用缓存，减少数据库压力。

### 小白解释

写入链路负责“记账”，聚合链路负责“算账”，查询链路负责“展示账单”。

### 可能追问

问：为什么有快照表？  
答：快照表用于保存历史榜单状态，比如某一小时 TOP100 是什么。这样可以回看历史排名、分析话题生命周期，也方便答辩时证明榜单不是临时拼出来的。

## 第 8 页：演示流程 6 步

### 推荐讲稿

现场演示可以按 6 步进行。第一，启动 MySQL 和 Redis。第二，启动 Spring Boot 后端。第三，打开 Vue 前端。第四，查看初始榜单。第五，模拟点赞、收藏、转发、评论。第六，触发聚合并刷新页面，观察榜单变化。

这个流程对应系统真实运行方式，也能让老师清楚看到：互动不是只写了数据库，而是真的影响了榜单。

### 小白解释

演示不要一开始就点很多按钮。先看原始榜单，再互动，再聚合，再刷新。这样老师才能看到“前后变化”。

### 可能追问

问：为什么演示时要手动触发聚合？  
答：实际系统可以定时每 5 分钟聚合。答辩现场为了不等待定时任务，用受 token 保护的运维接口手动触发一次聚合，逻辑和定时聚合一致。

## 第 9 页：数据库模型

### 推荐讲稿

数据库模型围绕榜单事实和用户行为设计。`circle` 保存圈子信息；`topic` 保存话题、当前热度分、互动数和状态；`interaction_event` 保存有效互动流水；`user_behavior` 保存有效和无效行为审计；`topic_score_snapshot` 保存历史榜单快照；`user_circle_preference` 保存用户对不同圈子的偏好，用于个性化榜单。

这里的设计对应 7 天文档第一天的数据库建模要求。

### 小白解释

可以记成三类表：内容表、行为表、结果表。  
内容表是 circle 和 topic。  
行为表是 interaction_event 和 user_behavior。  
结果表是 topic_score_snapshot 和 user_circle_preference。

### 可能追问

问：`interaction_event` 和 `user_behavior` 有什么区别？  
答：`interaction_event` 更关注有效互动，用来参与热度计算。`user_behavior` 更关注行为审计，包含有效、无效以及无效原因，用来解释反刷结果。

## 第 11 页：后端章节开场

### 推荐讲稿

下面进入后端部分。后端是整个项目的核心，因为热度算法、多维榜单、反刷、缓存、Redis 同步和定时聚合都在后端完成。我们会按 7 天实训文档的任务线来讲，重点不是把每一天平均介绍，而是讲每一天落地的核心能力。

### 小白解释

这一页提醒听众：后端部分要和 7 天文档对齐。老师如果按文档问，就能接上。

### 可能追问

问：7 天任务中最核心的是哪几天？  
答：Day1 到 Day4 是功能核心，Day5 到 Day7 是验证和交付。Day1 定模型和算法，Day2 做互动与多榜单，Day3 做反刷和缓存，Day4 做个性化、推送、屏蔽和重构。

## 第 12 页：Day1 建模和算法

### 推荐讲稿

Day1 主要完成业务建模和算法选型。我们先确定核心对象，包括圈子、话题、互动、用户行为、榜单快照和用户偏好。然后对比三种热度算法：纯计数、时间衰减和威尔逊置信区间。最终选择时间衰减作为主算法，因为它能同时考虑互动数量和内容新鲜度，更适合实时热点榜单。

公式可以概括为：热度分等于互动权重累计，除以发布时间距离当前的时间衰减，再乘上反刷倍数。

### 小白解释

纯计数就是看总分。时间衰减就是越新的内容越有机会。威尔逊更适合评价“质量稳定不稳定”，但当前项目主要目标是实时热点，所以选择时间衰减。

### 可能追问

问：为什么指数是 1.8？  
答：`1.8` 参考 Hacker News 类热榜公式经验值。指数越大，旧内容下降越快；指数越小，历史高互动内容越容易长期霸榜。课程项目先采用经验值，后续可以配置化或通过 A/B 实验调参。

## 第 13 页：互动权重

### 推荐讲稿

项目里不同互动类型权重不同。点赞权重是 1，收藏是 2，转发是 3，评论是 5。这个设计来自业务理解：点赞成本最低，收藏代表长期价值，转发代表传播价值，评论代表讨论深度，所以评论权重最高。

这也对应 7 天文档第二天的要求：设计热度聚合策略，不同互动类型采用不同权重。

### 小白解释

不要说“我们随便设的”。要说这是按用户行为成本和业务价值设计的默认权重。

### 可能追问

问：这些权重能不能改？  
答：可以。课程文档里提到权重配置可以支持热更新。当前项目用固定默认值落地，后续可以把权重放入配置表或配置中心。

## 第 14 页：时间衰减

### 推荐讲稿

时间衰减解决的是旧内容长期霸榜问题。公式里的分母是 `(publishHours + 2)^1.8`。话题发布越久，分母越大，如果没有新的高价值互动，热度分就会下降。`+2` 是为了避免刚发布内容年龄接近 0 时分数过高。

这样榜单不是历史总榜，而是更接近“现在正在热”的内容列表。

### 小白解释

可以把它理解成“热度会自然冷却”。内容越久，越需要更多新互动才能保持排名。

### 可能追问

问：如果旧内容持续有新评论，会不会还能上榜？  
答：会。时间衰减不是禁止旧内容上榜，而是要求旧内容必须持续获得足够互动才能抵消时间衰减。

## 第 15 页：Day2 互动、聚合和多榜单

### 推荐讲稿

Day2 主要把互动写入和多维榜单落地。用户的每次点赞、收藏、转发、评论会写入互动事件表。聚合任务读取这些事件，计算话题当前热度，更新 `topic.current_score`。在此基础上，系统提供全站榜、圈子榜、新人榜和飙升榜。

全站榜看全平台热点；圈子榜只看某个兴趣圈子；新人榜关注 24 小时内新发布内容；飙升榜关注最近 1 小时增长最快的话题。

### 小白解释

全站榜回答“全平台现在什么最热”。  
圈子榜回答“某个圈子里什么最热”。  
新人榜回答“新内容里谁表现好”。  
飙升榜回答“谁正在突然变热”。

### 可能追问

问：飙升榜和全站榜有什么区别？  
答：全站榜更看当前总热度，受历史互动影响；飙升榜更看短时间增量，更适合发现正在爆发的新话题。

## 第 16 页：窗口函数

### 推荐讲稿

窗口函数用于把排名计算放在数据库侧完成。比如圈子榜需要在每个 `circle_id` 内部独立排名，这就可以用 `RANK() OVER (PARTITION BY circle_id ORDER BY score DESC)`。`PARTITION BY` 表示按圈子分组，`ORDER BY score` 表示每组里按分数排序。

这样做的好处是数据库直接返回带排名的结果，Java 服务不用把所有数据查出来再手动排序。

### 小白解释

窗口函数可以理解为“数据库内置的分组排名工具”。它不是普通 `GROUP BY`，因为它不会把多行合并成一行，而是在保留每条记录的同时给它们计算排名。

### 可能追问

问：为什么不用 Java 排序？  
答：少量数据可以 Java 排序，但榜单查询是高频场景，把分组排名放在数据库侧可以利用索引和 SQL 优化，也减少应用层内存和网络传输。

## 第 17 页：Day3 反刷

### 推荐讲稿

Day3 重点是反刷机制。项目识别几类异常行为：同一用户短时间内重复互动，同一设备指纹对应多个账号集中互动，或者某话题短时间内异常突增。系统会根据风险强弱采取不同处理。强风险直接拒绝进入有效互动流水，弱风险通过 `weight_multiplier` 降低热度贡献，同时记录在用户行为表里。

这样既能防止刷量污染榜单，又能避免误伤正常用户。

### 小白解释

反刷不是“看到可疑就删掉”，而是分等级处理。强证据拒绝，弱证据降权。

### 可能追问

问：为什么设备指纹风险只降权，不直接拒绝？  
答：相同设备可能是家庭电脑、实验室机器或公共网络，不一定是刷量。直接拒绝可能误伤，所以采用降权更稳妥。

## 第 18 页：缓存策略

### 推荐讲稿

榜单是高频读取接口，如果每次都查 MySQL 并排序，数据库压力会比较大。所以项目使用缓存策略。本地缓存通过 `RankingCacheManager` 缓存常用榜单结果；空值缓存防止不存在数据反复打到数据库；Redis ZSet 用来维护排名集合，支持高频榜单读取和 MySQL 结果对比。

项目实测中，缓存预热后命中率达到 77.78%，Redis 同步后 MySQL 和 Redis 的全站 Top3 保持一致。

### 小白解释

缓存就是把常用结果先放在更快的地方。榜单读多写少，很适合缓存。

### 可能追问

问：缓存什么时候失效？  
答：热度聚合后、话题屏蔽或恢复后、互动写入影响个性化偏好后都会清理相关缓存。另外缓存本身也有短 TTL。

## 第 19 页：Day4 个性化、SSE、屏蔽和任务锁

### 推荐讲稿

Day4 是需求变更日，新增了个性化榜单、实时推送、话题屏蔽和遗留代码重构。个性化榜单通过 `user_circle_preference` 记录用户对不同圈子的偏好，在全站候选集基础上重排。SSE 用于榜单更新事件推送。话题屏蔽通过更新 `topic.status` 并清理缓存，让内容立即从榜单消失。任务锁用于避免多实例定时任务重复执行。

### 小白解释

个性化榜单解决“千人一面”。SSE 解决“服务端主动通知前端”。话题屏蔽解决“运营下架内容”。任务锁解决“多个后端实例重复跑同一个任务”。

### 可能追问

问：个性化榜单和全站榜单查询有何本质区别？  
答：全站榜只依赖全局热度，缓存 key 只和榜单类型、limit 有关。个性化榜还依赖 `userId` 和用户圈子偏好，同一个话题对不同用户可能排序不同，所以缓存 key 必须包含用户 ID。

## 第 20 页：接口分类

### 推荐讲稿

后端接口可以分为四大类。榜单查询接口返回全站、圈子、新人、飙升和个性化榜单。互动写入接口接收用户行为。分析接口给前端 Dashboard 提供概览数据。运维接口用于手动触发热度聚合、快照和 Redis 同步。

这些接口都使用统一响应结构 `ApiResponse`，也就是包含 `code`、`message` 和 `data`。

### 小白解释

统一响应结构的好处是前端和 Apifox 测试都容易判断成功或失败。

### 可能追问

问：运维接口为什么要 token？  
答：聚合和快照属于高权限操作，不能让任意用户触发。课程项目里用固定 token 演示，真实生产环境应接入权限系统。

## 第 28 页：接口契约

### 推荐讲稿

接口测试重点不是软件本身，而是接口契约是否成立。比如榜单接口要返回排名、话题 ID、标题、分数、互动数；互动写入接口要能处理正常请求和异常请求；运维聚合接口要校验 token；分析接口要返回前端可用的统计字段；反刷报告要返回无效行为数量和原因分布。

### 小白解释

接口契约就是“前端和后端说好的格式”。如果后端返回字段不对，前端就可能渲染失败。

### 可能追问

问：接口测试主要断言什么？  
答：断言 HTTP 状态码、业务 code、message、data 结构、关键字段是否存在，以及异常场景是否按预期返回错误。

## 第 35 页：JMeter 压测覆盖接口

### 小白解释

只读压测就是不写新数据，主要看系统在大量查询时响应快不快、会不会报错。

## 3. 按 7 天文档准备追问

### Day1：业务建模和热度算法

问：热榜业务本质是什么？  
答：热榜不是简单统计谁互动多，而是找出当前最值得关注的内容。它要同时考虑互动强度、内容新鲜度、互动真实性和圈子差异。

问：三种算法怎么取舍？  
答：纯计数简单但缺乏时效性；时间衰减能兼顾互动强度和发布时间，适合实时热榜；威尔逊置信区间适合质量榜，因为它关注正反馈比例和样本量稳定性。我们最终用时间衰减作为主算法，威尔逊可以作为后续质量辅助分。

问：数据库建模为什么需要这些表？  
答：`topic` 存当前话题和热度，`interaction_event` 存有效互动流水，`user_behavior` 存反刷审计，`topic_score_snapshot` 存历史榜单，`circle` 支持圈子榜，`user_circle_preference` 支持个性化榜单。

### Day2：互动采集、聚合和榜单

问：为什么用定时聚合？  
答：互动写入是高频事件，如果每次互动都立即全量重算榜单，会造成较大数据库和计算压力。定时聚合把事件批量计算成话题热度，符合分钟级热榜需求。

问：快照机制有什么用？  
答：快照可以保存历史 TOP 榜单，用于回溯、分析趋势、排查问题，也能证明榜单是持续生成的。

问：窗口函数具体用在哪里？  
答：用于数据库侧排名，特别是圈子榜，可以按 `circle_id` 分区并在每个圈子内部按分数排序。

问：索引为什么重要？  
答：榜单查询是高频读场景，如果没有合适索引，每次排序和过滤都会扫描大量数据。复合索引能让 `status + score`、`circle_id + status + score` 这类查询更快。

### Day3：防刷和缓存

问：同一用户 24 小时互动上限为什么要设？  
答：这是滑动窗口频率限制，用来防止同一用户反复刷同一话题。超过上限的行为不计入热度，但保留日志。

问：缓存穿透和缓存雪崩是什么？  
答：缓存穿透是反复查询不存在的数据，导致请求一直打到数据库；空值缓存可以缓解。缓存雪崩是大量缓存同一时间过期，导致数据库瞬间被打满；随机 TTL 可以把过期时间打散。

问：缓存和数据库不一致怎么办？  
答：当前通过短 TTL、聚合后主动清理、屏蔽后主动失效、Redis 同步来控制不一致窗口。热点榜单允许近实时，不要求每次互动毫秒级全局一致。

### Day4：需求变更和重构

问：个性化榜单怎么实现？  
答：记录用户在不同圈子的互动偏好，然后在全站候选榜单基础上按用户偏好重排。这样避免为每个用户全库排序。

问：SSE 是什么？  
答：SSE 是 Server-Sent Events，服务端可以通过长连接向浏览器推送事件。项目里可用于榜单更新或话题进入 TOPN 的通知。

问：话题屏蔽为什么能立即生效？  
答：榜单查询过滤 `status=1` 的正常话题。屏蔽接口更新状态并清理缓存，下一次查询就不会出现该话题。

问：多实例定时任务为什么需要锁？  
答：如果部署多个后端实例，每个实例都可能触发聚合任务。MySQL named lock 能保证同一时间只有一个实例执行，避免重复聚合和重复快照。

### Day5：压测和数据分析

问：7 天文档要求混合压测，你们现在做到什么程度？  
答：当前已经完成 JMeter 只读核心接口压测，三轮错误率都是 0，吞吐稳定在 6500 req/s 以上。同时通过互动模拟和内置轻量测试验证写入链路。混合压测是后续增强方向，可以同时压 500 TPS 写入和 1000 QPS 读取。

问：压测结果说明瓶颈在哪里？  
答：只读榜单接口很轻，P99 约 8 到 9ms；分析概览相对更重，P99 约 27 到 28ms。更大规模下，写入事件表、全量聚合扫描和分析接口会成为优先优化对象。

问：如果互动量增长 10 倍，怎么演进？  
答：用 Kafka 承接互动写入峰值，异步消费入库和聚合；Redis ZSet 升级为主读链路；MySQL 做审计和离线分析；必要时做读写分离和冷热数据分层。

### Day6：文档、Demo 和互评

问：Demo 三个场景是什么？  
答：第一，实时互动触发热度变化并反映到榜单。第二，刷热度行为被拦截或降权并记录。第三，话题屏蔽后立即从榜单消失。

问：文档有哪些？  
答：README、接口参考文档、系统架构说明、数据库设计说明、性能与数据分析说明、演示手册、互动模拟记录、第七天答辩手册等。

问：互评重点应该看什么？  
答：缓存使用是否正确，事务边界是否合理，防刷逻辑是否有漏洞，SQL 和索引是否匹配查询场景，接口文档是否和实现一致。

### Day7：答辩和复盘

问：如果一个话题同时被 1000 个用户点赞，数据库能不能扛住？  
答：课程项目中，互动会作为事件流水写入 MySQL，配合连接池和索引可以支撑一定并发。但真正大规模场景下，写入流水会成为瓶颈。后续会引入 Kafka 削峰，异步消费入库和聚合，并把 Redis ZSet 作为主读链路。

问：缓存和数据库不一致窗口业务能接受吗？  
答：热点榜单业务通常接受近实时，不要求强一致。当前通过短 TTL、聚合后失效、Redis 同步缩短窗口。相比每次互动立即全局重算，近实时方案更适合性能和工程复杂度的平衡。

问：你们和操作系统、软件工程课程有什么关联？  
答：定时任务和多实例锁可以类比操作系统里的进程调度和互斥；缓存策略、接口文档、测试报告、需求变更和重构则对应软件工程里的系统设计、质量保证和维护演进。
