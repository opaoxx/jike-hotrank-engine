## 1. 测试目标

本次测试用于支撑答辩 PPT 中的“Apifox 接口测试”章节，重点验证三件事：

1. 后端核心接口是否能按文档被正常调用。
2. 正常链路、异常链路和运维链路是否都有返回结果。
3. 接口实现和接口文档之间是否存在契约差异。

这里的“契约”指前后端、测试工具和文档共同约定的内容，包括 HTTP 状态码、业务 `code`、`message`、`data` 结构和关键字段。

## 5. 本次发现的问题

### 5.1 HTTP 状态码与接口文档不一致

接口参考文档中描述：

- 参数错误应返回 HTTP `400`。
- 话题不存在应返回 HTTP `404`。
- 鉴权失败应返回 HTTP `403`。

本次实测发现，当前实现对这些业务错误统一返回 HTTP `200`，再通过响应体中的 `code` 表示错误类型。

影响：

- 前端如果只看 `code`，功能上没有问题。
- Apifox 如果断言 HTTP 状态码，会出现失败。
- 接口文档和实现不完全一致，答辩时可能被老师追问。

建议：

1. 更推荐修全局异常处理，让 HTTP 状态码和业务 code 保持一致。
2. 如果暂时不修改代码，则应调整接口参考文档，明确当前项目使用“HTTP 200 + 业务 code”的统一响应策略。

答辩回答建议：

> 我们测试时发现异常场景的业务 code 是正确的，比如不存在话题返回 code=404，非法互动返回 code=400，错误 token 返回 code=403。但当前全局响应封装没有把这些业务错误映射到 HTTP 状态码，所以 HTTP 层仍然是 200。这个属于接口契约一致性问题，不影响前端按 code 判断，但后续应统一文档和实现。

### 5.2 SSE 长连接未纳入本次自动脚本

`GET /api/notifications/rankings/stream` 是 SSE 长连接接口，不适合放入普通短请求脚本中直接等待结束。

建议在 Apifox 或浏览器中单独验证：

```javascript
const es = new EventSource('/api/notifications/rankings/stream');
es.addEventListener('connected', e => console.log(e.data));
es.addEventListener('ranking-updated', e => console.log(e.data));
```

答辩时可以说明：SSE 属于持续连接，测试方式和普通 REST 接口不同。

## 6. Apifox 复现建议

建议在 Apifox 中建立 4 个分组：

| 分组 | 用例 |
| --- | --- |
| 榜单查询 | 全站榜、圈子榜、新星榜、飙升榜、个性化榜 |
| 互动写入 | 正常互动、非法互动类型、话题不存在、频率限制 |
| 分析与反刷 | 分析总览、热度分布、互动统计、圈子活跃度、反刷报告 |
| 运维与 Redis | 聚合、快照、Redis 同步、Redis 全站榜、错误 token |

建议断言：

1. HTTP 状态码。
2. 响应体 `code`。
3. 响应体 `message`。
4. 响应体 `data` 是否存在。
5. 榜单接口的 `data.items` 是否为数组。
6. 榜单条目是否包含 `rank`、`topicId`、`title`、`score`、`interactionCount`。

当前如果按接口文档断言 HTTP `400/403/404`，异常用例会失败；如果按业务 `code` 断言，则异常处理逻辑通过。

## 7. 答辩讲法

可以这样讲：

> 我们用 Apifox 思路设计了接口测试集合，覆盖榜单查询、互动写入、分析反刷、运维和 Redis 接口。本地实测 24 个用例，其中正常链路全部可用，Redis 和聚合接口也可用。测试中发现 3 个异常场景存在接口契约差异：业务 code 正确，但 HTTP 状态码统一为 200。这个问题不影响前端按业务 code 判断，但说明接口文档和全局异常处理需要进一步统一。

如果老师问“测试发现问题了吗”，不要说没有，可以答：

> 有。我们发现异常响应的业务语义是正确的，但 HTTP 状态码和文档不一致。这说明接口测试不只是证明功能能跑，也能发现文档和实现的契约差异。

## 8. 后续改进

1. 统一异常响应策略：让 `BusinessException` 映射到对应 HTTP 状态码，或者在文档中明确统一 HTTP 200 策略。
2. 为 Apifox 补充截图：接口集合、运行结果、断言失败示例、正常接口响应示例。
3. 增加 SSE 专项测试说明。
4. 增加频率限制专项测试：同一用户短时间内重复互动，验证 `user_behavior` 审计记录。
5. 将接口测试集合导出为 Apifox 项目文件，方便组员复现。
