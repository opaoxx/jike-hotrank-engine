# 窗口函数实践说明

本文档对应 Day2 的 “MySQL 窗口函数实践：`RANK() OVER (PARTITION BY circle_id ORDER BY score DESC)`” 要求。

## 业务背景

线上接口 `/api/ranking/circle/{circleId}` 查询的是单个圈子的 TOPN：

```sql
SELECT id, circle_id, title, content, author_id, publish_time, current_score,
       interaction_count, status, created_at, updated_at
FROM topic
WHERE circle_id = ?
  AND status = 1
ORDER BY current_score DESC
LIMIT ?;
```

这个查询能直接利用 `idx_circle_status_score (circle_id, status, current_score DESC)`，适合线上高频请求。

窗口函数更适合一次性查看所有圈子内排名，例如报表、教学演示、离线分析：

```sql
SELECT *
FROM (
    SELECT id, circle_id, title, current_score,
           RANK() OVER (PARTITION BY circle_id ORDER BY current_score DESC) AS circle_rank
    FROM topic
    WHERE status = 1
) ranked
WHERE circle_rank <= 20
ORDER BY circle_id, circle_rank;
```

## 和应用层排序的对比

| 方案 | 优点 | 缺点 | 当前定位 |
| --- | --- | --- | --- |
| 单圈 SQL + `LIMIT` | 命中复合索引，返回行数少，适合接口 | 每次只能查一个圈子 | 线上接口主方案 |
| 窗口函数 | 一次得到所有圈子内排名，SQL 表达力强 | 参与排序的数据更多，可能产生临时表或 filesort | 报表/教学/分析方案 |
| 应用层排序 | Java 逻辑灵活，便于组合个性化权重 | 需要拉更多数据到内存，数据库无法提前裁剪 | 个性化候选重排使用 |

## EXPLAIN 检查点

执行：

```sql
EXPLAIN SELECT *
FROM (
    SELECT id, circle_id, title, current_score,
           RANK() OVER (PARTITION BY circle_id ORDER BY current_score DESC) AS circle_rank
    FROM topic
    WHERE status = 1
) ranked
WHERE circle_rank <= 20
ORDER BY circle_id, circle_rank;
```

重点观察：

- `key` 是否使用 `idx_status_score`、`idx_circle_status_score` 或其他与 `status/circle_id/current_score` 相关的索引。
- `rows` 是否接近整表规模。
- `Extra` 中如果出现 `Using temporary` 或 `Using filesort`，需要结合数据量判断是否可接受。

## 结论

项目保留线上单圈榜单查询作为主实现，确保接口路径简单、低延迟、容易缓存；同时用窗口函数 SQL 覆盖课程中的 MySQL Window Function 实践要求，用于答辩时解释数据库排序能力和应用层排序的取舍。
