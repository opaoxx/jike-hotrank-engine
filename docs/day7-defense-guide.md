# Day7 答辩问答手册

本文档用于项目答辩前快速复盘。回答原则是先给结论，再讲工程取舍，最后指出后续演进方向。

## 讲解主线

1. 业务问题：内容社区需要把用户互动在分钟级内反映到热点榜单。
2. 核心链路：互动写入、反作弊、热度聚合、榜单缓存、实时推送。
3. 工程难点：热度算法合理性、缓存一致性、防刷误伤、多实例任务、数据库瓶颈。
4. 演进方向：Redis ZSet、Kafka 削峰、读写分离、离线分析。

## 高频追问

### 1. 时间衰减指数 1.8 是怎么确定的？

答辩回答：

`1.8` 来自 Hacker News 类热榜公式的经验值。它让互动分随时间快速但不过度地衰减：新内容能获得曝光窗口，旧内容如果没有持续互动会逐步下降。指数越大，榜单越偏向新内容；指数越小，历史高互动内容越容易长期霸榜。

本项目把它作为课程项目默认值，而不是写死在 SQL 中，后续可以通过配置化或 A/B 实验按业务目标调整。

### 2. 为什么不用纯互动数排序？

答辩回答：

纯互动数只表达“总量”，不能表达“现在是否正在热”。社区热榜关注的是当前热度，如果只按总量排序，早期积累大量互动的旧话题会长期占据榜单。时间衰减模型能兼顾互动质量和时效性，更符合热点榜单业务。

### 3. 评论、转发、收藏、点赞为什么权重不同？

答辩回答：

不同互动代表不同用户投入成本。点赞成本最低，所以权重为 1；收藏表示用户认为内容有长期价值；转发代表内容有传播价值；评论需要更高参与成本，也更能体现讨论热度，因此权重最高。

当前默认权重是点赞 1、收藏 2、转发 3、评论 5，后续可以放入配置中心或数据库热更新。

### 4. 缓存和数据库不一致窗口有多长？

答辩回答：

当前榜单缓存是 JVM 本地缓存，TTL 为秒级并带随机抖动。更重要的是，系统在热度聚合、话题屏蔽、话题恢复等关键写路径后主动清理榜单缓存。因此不一致窗口主要来自两类情况：聚合周期内互动还没有转化为 `topic.current_score`，以及短暂的缓存 TTL 窗口。

课程项目中，这个窗口符合热点榜单“近实时”要求；如果需要强一致，应改为同步更新榜单结构或使用分布式缓存事件失效。

### 5. 如果 1000 个用户同时点赞一个话题，数据库能不能扛住？

答辩回答：

当前实现会把每次互动作为事件流水写入 MySQL，课程级并发可以通过连接池、索引和批量聚合支撑。真正到高并发时，最先遇到瓶颈的是写入流水和聚合查询，而不是榜单读取，因为读取已经有缓存。

演进方案是：互动入口先写 Kafka，异步消费入库和聚合；热点榜单用 Redis ZSet 承接实时排序；MySQL 保留为审计和离线分析数据源。

### 6. 个性化榜单和全站榜单在查询上有什么本质区别？

答辩回答：

全站榜单只依赖话题全局热度，缓存 key 只和榜单类型、limit 有关。个性化榜单还依赖用户圈子偏好，同一个话题对不同用户的排序可能不同，所以缓存 key 必须包含 `userId`，并在用户偏好变化后清理该用户的个性化缓存。

本项目采用“全站候选集 + 用户偏好重排”的方式，避免为每个用户直接全库排序。

### 7. 防刷为什么不是把可疑互动全部拒绝？

答辩回答：

防刷信号强弱不同。频率超限属于强信号，系统直接拒绝进入有效互动流水；设备指纹多账号属于弱信号，可能存在家庭、实验室或网吧共享设备，直接拒绝容易误伤。因此系统采用降权方式，通过 `weight_multiplier` 降低热度贡献，同时保留审计记录。

### 8. 多实例部署时定时任务会不会重复跑？

答辩回答：

定时聚合和榜单快照都通过 MySQL named lock 做互斥。每个实例触发任务时先获取同一个逻辑锁，只有拿到锁的实例执行任务，其他实例跳过。这样避免重复聚合、重复生成快照。

### 9. 话题屏蔽为什么能立即从榜单消失？

答辩回答：

榜单查询只返回 `status=1` 的正常话题。屏蔽接口会把话题状态更新为屏蔽，并立即清理榜单缓存。下一次榜单查询会从数据库重新加载，屏蔽话题不会再进入结果。

### 10. 项目最大的瓶颈和下一步优化是什么？

答辩回答：

当前最大瓶颈在互动事件写入和全量聚合。读取路径已经有缓存，短期主要压力在 MySQL 写入、聚合扫描和单机本地缓存的多实例一致性。下一步优先做三件事：Kafka 削峰、Redis ZSet 实时榜单、分析库或读写分离承接报表查询。

## Demo 讲解词

### 互动触发榜单变化

“现在我先查询全站热榜，然后给 `topicId=1` 写入评论和转发。评论权重是 5，转发权重是 3。为了答辩现场不等待 5 分钟定时任务，我用受 token 保护的运维入口手动触发一次聚合。聚合完成后可以看到话题热度和榜单位置变化，同时 SSE 订阅能收到榜单更新事件。”

### 防刷拦截

“这里模拟同一用户对同一话题短时间内连续互动。系统会把超过频率限制的行为写入 `user_behavior` 审计表，但不会写入有效互动流水，因此不会继续贡献热度。设备指纹风险则采用降权，减少误伤。”

### 话题屏蔽

“运营屏蔽话题时，系统更新 `topic.status`，同时清理榜单缓存。再次查询榜单时，屏蔽话题立刻消失。演示结束后我再恢复话题并重新聚合。”

## 结尾复盘

本项目完成了从业务建模到性能分析的完整链路。它不是简单 CRUD，而是围绕热点榜单这个高频读、持续写、需要反作弊和近实时更新的核心场景，做了算法、缓存、事务、索引和可运维性的综合实现。
