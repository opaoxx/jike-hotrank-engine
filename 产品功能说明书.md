## 1. 产品概述

即刻 App 内容社区实时热点榜单引擎，是一个面向内容社区的热点发现与榜单管理系统。

它围绕“话题、圈子、互动、热度、防刷、榜单展示”建立完整链路。用户在社区中点赞、收藏、转发、评论后，系统会采集互动事件，根据互动权重、发布时间和风险降权计算话题热度，再生成不同维度的榜单。

本系统适合用于课程实训、产品原型展示、运营后台 Demo 和热点榜单算法讲解。它不是一个完整的社交 App，而是一个独立的“热点榜单引擎”和“榜单运营看板”。

系统已经提供后端服务、数据库脚本、前端管理面板、缓存与 Redis 对比、反作弊审计、压测入口、API 文档和答辩材料。

### 1.1 产品解决的问题

内容社区的运营人员通常会遇到几个问题：

- 全站现在最热门的话题是什么？
- 某个兴趣圈子里哪些内容正在变热？
- 新发布的话题是否有机会进入推荐视野？
- 哪些话题在短时间内突然升温？
- 用户互动是否被刷量行为影响？
- 榜单查询是否能在高频访问下保持稳定？

本系统把这些问题拆成可查看、可操作、可验证的功能模块，让老师或用户可以通过网站直接观察榜单变化。

### 1.2 一句话定位

本系统是一个“可互动、可聚合、可防刷、可展示、可分析”的内容社区热点榜单引擎。

## 3. 系统入口与整体使用流程

### 3.1 使用入口

系统默认后端地址为：

```text
http://localhost:8080
```

前端构建后由 Spring Boot 静态资源服务提供访问：

```text

```


### 3.2 典型演示流程

用户打开系统后，可以按下面顺序理解产品：

1. 进入总览页，查看当前话题数、互动数、热度分布、圈子活跃度、防刷数据和缓存命中率。
2. 进入全站热榜，查看整个社区最热的话题。
3. 进入圈子热榜，选择科技圈、游戏圈、美食圈等圈子，观察局部热门内容。
4. 进入新星榜，查看 24 小时内发布并获得热度的新话题。
5. 进入飙升榜，查看最近 1 小时升温最快的话题。
6. 点击榜单中的话题，查看详情，并进行屏蔽或恢复操作。
7. 使用底部互动模拟器提交点赞、收藏、转发、评论。
8. 等待定时聚合，或通过运维接口手动触发聚合。
9. 回到榜单页面，观察话题分数、互动数和排名变化。
10. 进入反作弊页面，查看可疑行为、拦截原因和审计统计。

这条流程可以完整展示“用户互动产生数据，后端聚合形成热度，前端榜单发生变化，防刷模块提供保护”的闭环。

## 5. 前端页面功能说明

前端基于 Vue 3、Vue Router、Pinia、Element Plus 和 ECharts 构建，共 6 个主要页面，并在底部提供互动模拟器。

### 5.1 总览页面

总览页面是系统首页，用来快速了解社区整体运行状态。

页面展示内容包括：

- 热度分布柱状图：查看话题分数集中在哪些区间。
- 互动类型饼图：查看点赞、收藏、转发、评论的占比。
- 圈子活跃雷达图：比较不同圈子的话题数量、平均热度和互动数。
- 缓存命中率仪表盘：观察榜单缓存是否生效。
- 防刷概览：查看近期无效行为和拦截趋势。

这个页面适合答辩开场展示，让观众先知道系统目前有数据、有榜单、有统计、有缓存状态。

### 5.2 全站热榜页面

全站热榜展示整个社区当前热度最高的话题。

用户可以看到每条话题的排名、标题、圈子、热度分、互动数、作者和发布时间。

点击榜单行后，系统会弹出话题详情窗口。运营人员可以在详情中查看内容，并执行屏蔽或恢复操作。

### 5.3 圈子热榜页面

圈子热榜用于查看某个兴趣圈内部的热门话题。

例如科技圈、游戏圈、美食圈、旅行圈、读书圈可以分别形成自己的局部榜单。这样可以避免全站热门内容长期占据视野，让小圈子内的高质量内容也能被发现。

### 5.4 新星榜页面

新星榜关注最近 24 小时内发布的新话题。

系统会先筛选正常状态且发布时间在 24 小时内的话题，再按当前热度分排序。它的产品价值是给新内容一个展示机会，避免老话题长期占据全部曝光。

### 5.5 飙升榜页面

飙升榜关注短时间内互动增长明显的话题。

系统会比较最近 1 小时和前 1 小时的加权互动情况，再按飙升分排序。它适合发现突然升温的内容，比如突发讨论、热点新闻、短时间集中传播的话题。

### 5.6 反作弊页面

反作弊页面用于展示系统对可疑互动的识别结果。

页面内容包括：

- 拦截趋势折线图。
- 拦截原因分布图。
- 可疑用户列表。
- 受影响话题数量。
- 不同互动类型下的异常统计。

这部分证明系统不是只会“加分排序”，还会考虑刷榜和异常行为带来的榜单污染。

### 5.7 互动模拟器

互动模拟器位于前端底部，可以直接向后端提交互动事件。

用户可以输入话题 ID、用户 ID、互动类型、设备指纹和 IP 地址。提交后，后端会完成参数校验、话题检查、防刷判断、行为审计、互动写入和偏好更新。

这个组件非常适合答辩演示，因为它能现场制造互动数据，再观察榜单变化。

## 6. 榜单规则与热度计算

### 6.1 榜单类型

系统提供 5 种榜单。

| 榜单 | 说明 | 适用场景 |
| --- | --- | --- |
| 全站热榜 | 全社区按热度排序 | 展示社区总热点 |
| 圈子热榜 | 指定圈子内按热度排序 | 展示某个兴趣圈热门内容 |
| 新星榜 | 24 小时内新话题按热度排序 | 发现新发布的潜力内容 |
| 飙升榜 | 最近 1 小时升温最快的话题 | 发现短时间突增热点 |
| 个性化榜单 | 在全站候选池上按用户圈子偏好重排 | 模拟个性化推荐效果 |

### 6.2 互动权重

系统将不同互动视为不同强度的热度信号。

| 互动类型 | 编码 | 默认权重 | 产品含义 |
| --- | ---: | ---: | --- |
| 点赞 | 1 | 1 | 轻量认可 |
| 收藏 | 2 | 2 | 用户认为内容有保存价值 |
| 转发 | 3 | 3 | 用户愿意传播内容 |
| 评论 | 5 | 5 | 用户投入更多表达成本 |

权重可以通过 `application.yml` 配置调整，不需要修改代码。

### 6.3 风险降权

系统不会把所有可疑互动都简单丢弃。

对于频率超限的互动，系统会拒绝写入有效互动事件。对于设备指纹命中多账号风险的互动，系统会保留事件，但通过 `weight_multiplier` 降低热度贡献。

聚合 SQL 的核心思想是：

```text
每条互动贡献 = 原始互动权重 × 风险降权系数
话题加权互动分 = 同一 topic_id 下所有互动贡献求和
```

例如，一条评论的原始权重是 5。如果设备风险降权系数是 0.5，那么这条互动对热度的贡献就是 2.5。

### 6.4 热度分公式

系统最终使用时间衰减公式计算话题热度：

```text
score = weightedInteractionScore / (publishHours + 2)^1.8
```

其中：

- `weightedInteractionScore` 是互动权重和风险降权后的总分。
- `publishHours` 是话题发布到当前的小时数。
- `+2` 用来避免新话题发布时间太短导致分数异常。
- `1.8` 表示热度会随时间逐渐衰减。

这个公式体现了两个原则：

- 互动越多、互动越重，热度越高。
- 发布时间越久，旧内容会逐渐让位给新内容。

### 6.5 热度聚合任务

系统每 5 分钟执行一次热度聚合任务。

聚合流程如下：

1. 使用 MySQL named lock 获取任务锁，避免多实例重复执行。
2. 从 `interaction_event` 表按话题聚合加权互动分。
3. 使用 `HeatScoreCalculator` 结合发布时间计算当前热度分。
4. 批量更新 `topic.current_score` 和 `topic.interaction_count`。
5. 清理榜单缓存，让下一次查询读到新分数。
6. 同步 Redis ZSet 榜单数据。
7. 通过 SSE 推送榜单更新事件。
8. 检测是否有新话题进入 Top 10。

这说明系统的榜单不是静态表格，而是由互动数据周期性驱动更新。

## 7. 互动采集与防刷机制

### 7.1 互动写入链路

用户提交一次互动后，后端会执行完整业务链路。

```text
参数校验
-> 查询话题是否存在
-> 判断话题是否可互动
-> 反作弊检查
-> 写入 user_behavior 审计记录
-> 写入 interaction_event 有效互动事件
-> 更新 user_circle_preference 用户圈子偏好
-> 清理该用户个性化榜单缓存
-> 检查异常突增
```

互动写入使用 `@Transactional` 管理事务。有效互动、用户行为记录、偏好更新等数据库操作处在同一个业务链路中。如果中间发生异常，事务会回滚，避免只写入一部分数据造成状态不一致。

### 7.2 三层防刷策略

系统内置三层防刷策略。

| 防刷策略 | 判断方式 | 处理结果 |
| --- | --- | --- |
| 频率限制 | 同一用户对同一话题 24 小时内互动超过阈值 | 拒绝互动，记录无效行为 |
| 设备指纹降权 | 同一设备关联多个用户 | 允许写入，但降低热度贡献 |
| 异常突增检测 | 最近 1 小时互动量达到最低数量，并超过历史小时均值 10 倍 | 将话题标记为待审核 |

默认配置为：

- 单用户单话题 24 小时内最多 50 次互动。
- 同设备关联用户数达到 3 个时，热度系数降为 0.5。
- 同设备关联用户数达到 5 个时，热度系数降为 0.3。
- 异常突增检测要求最近 1 小时互动量至少达到 10 条，并超过历史平均小时互动量 10 倍。

### 7.3 行为审计

系统使用 `user_behavior` 表记录用户行为审计。

有效互动和无效互动都会留下记录。无效互动会记录 `invalid_reason`，方便后续统计原因、排查异常和向老师解释防刷逻辑。

### 7.4 话题状态

话题状态分为 3 类。

| 状态 | 含义 | 榜单表现 |
| ---: | --- | --- |
| 0 | 已屏蔽 | 不出现在榜单中 |
| 1 | 正常 | 可以参与榜单排序 |
| 2 | 待审核 | 不作为正常话题进入榜单 |

运营人员可以通过话题详情对话题进行屏蔽或恢复。

## 8. 实时更新、缓存与 Redis 对比

### 8.1 榜单缓存

系统使用 JVM 本地缓存加速高频榜单读取。

榜单缓存基础 TTL 为 5 秒，并加入约 500 毫秒随机抖动。随机抖动可以把不同缓存 key 的过期时间打散，避免大量缓存同一时刻失效后请求同时打到数据库。

这类问题通常叫缓存雪崩。本项目通过随机 TTL 降低瞬时数据库压力。

### 8.2 空值缓存

如果某个榜单暂时没有数据，系统不会每次都穿透到数据库查询。

项目会通过 `putNull` 缓存空结果，TTL 大约为正常缓存的一半，并同样带有抖动。这样可以避免不存在或暂时为空的数据被反复请求，形成缓存穿透。

### 8.3 主动失效

当热度聚合完成、话题被屏蔽或恢复、个性化偏好发生变化时，系统会主动清理相关缓存。

这种策略保证了缓存既能减少数据库压力，又不会长期返回旧榜单。

### 8.4 Redis ZSet 对比方案

除 MySQL 查询链路外，系统还实现了 Redis Sorted Set 榜单对比方案。

Redis 中维护：

```text
ranking:global
ranking:circle:{circleId}
```

热度聚合任务会把话题分数同步到 Redis。用户可以通过 `/api/redis-ranking/` 前缀的接口查看 Redis 版本榜单。

当前项目中，Redis 主要用于对比和演示，不完全替代主榜单链路。主榜单接口仍以 MySQL 结果加本地缓存为准。

### 8.5 SSE 实时推送

系统提供 SSE 事件流：

```text
GET /api/notifications/rankings/stream
```

前端连接后，可以接收以下事件：

| 事件 | 触发时机 |
| --- | --- |
| `connected` | 建立连接 |
| `ranking-updated` | 热度聚合完成 |
| `top-n-entered` | 新话题进入 Top N |

这样前端可以在榜单更新时给出提示，而不需要一直高频轮询。

## 9. 运营与演示能力

### 9.1 话题详情

用户在榜单中点击话题后，可以查看：

- 话题标题。
- 话题内容。
- 所属圈子。
- 作者 ID。
- 当前热度分。
- 互动数。
- 发布时间。
- 当前状态。

运营人员可以据此判断话题是否应该保留在榜单中。

### 9.2 屏蔽与恢复

系统支持屏蔽和恢复话题。

屏蔽后，话题状态变为 `0`，并从全站榜、圈子榜、新星榜、飙升榜等正常榜单视图中移除。

恢复后，话题状态变为 `1`，并在下一次查询或聚合后重新参与榜单排序。

### 9.3 手动聚合与快照

除了定时任务，系统还提供手动触发能力。

| 操作 | 用途 |
| --- | --- |
| 手动热度聚合 | 演示时立即看到互动后的榜单变化 |
| 手动快照 | 立即记录当前 Top 100 榜单快照 |
| Redis 同步 | 把 MySQL 话题分数同步到 Redis ZSet |
| Redis 清空 | 清理 Redis 对比榜单数据 |

这些能力适合课程答辩和本地演示，不等同于生产环境的完整权限后台。

## 10. 数据分析与质量保障

### 10.1 分析总览

系统提供多个只读分析接口，前端总览页会组合展示这些数据。

| 分析项 | 说明 |
| --- | --- |
| 热度分布 | 统计不同分数区间的话题数量 |
| 互动统计 | 统计点赞、收藏、转发、评论数量和占比 |
| 圈子活跃度 | 比较不同圈子的话题数、平均热度和互动总数 |
| 防刷统计 | 查看无效行为原因、趋势和可疑对象 |
| 缓存统计 | 查看缓存条数、命中数、未命中数和命中率 |

### 10.2 压测与性能观察

项目提供两类压测方式：

- 内置轻量压测接口，适合本地快速验证。
- 文档中的外部压测脚本，适合生成更独立的测试结果。

默认内置压测接口的安全上限为：

- QPS：1 到 200。
- 持续时间：1 到 30 秒。
- 最大线程数：64。

因此，产品文档中不把“1000 QPS”写成当前系统绝对承诺。更准确的说法是：系统具备缓存、索引和 Redis 对比能力，支持进一步压测验证；具体吞吐量应以实际部署环境、数据量和压测报告为准。

### 10.3 数据库设计

系统核心业务表包括：

| 表 | 用途 |
| --- | --- |
| `circle` | 圈子信息 |
| `topic` | 话题内容、热度分、互动数和状态 |
| `interaction_event` | 有效互动事件 |
| `user_behavior` | 用户行为审计和无效原因 |
| `topic_score_snapshot` | 榜单快照 |
| `user_circle_preference` | 用户圈子偏好 |

数据库围绕高频访问设计复合索引，包括全站榜排序、圈子榜排序、新星榜筛选、互动聚合、设备指纹检测和防刷审计。

这些索引的目标是减少扫描范围和排序成本，而不是简单堆砌索引。

## 11. 系统边界与已知限制

### 11.1 包含范围

本系统包含：

- 话题和圈子数据模型。
- 点赞、收藏、转发、评论互动采集。
- 多维榜单查询。
- 热度聚合和时间衰减算法。
- 频率限制、设备指纹降权、异常突增检测。
- 榜单缓存、空值缓存、随机 TTL。
- Redis ZSet 对比榜单。
- SSE 榜单事件推送。
- 前端管理看板。
- 运维演示接口。
- 数据分析和压测入口。

### 11.2 不包含范围

本系统不包含：

- 用户注册和登录。
- 完整权限系统。
- 内容发布编辑后台。
- 移动端 App。
- IM 聊天。
- 搜索系统。
- AI 内容审核。
- 大规模消息队列。
- 多机房容灾。
- 生产级配置中心。

### 11.3 已知限制

当前项目是课程实训级实现，重点在功能闭环和架构理解。

已知限制包括：

- 热度聚合当前偏批处理思路，数据量极大时需要增量化或流式计算。
- Redis 当前主要是对比链路，尚未完全替代 MySQL 主榜单。
- 运维接口使用简单 token，生产环境应接入正式权限体系。
- 个性化榜单采用候选集重排，未实现完整推荐召回系统。
- 压测结果受本机环境、数据规模和 Redis/MySQL 状态影响，需要结合实际报告解释。

## 附录 B：部署运行

### B.1 环境要求

| 环境 | 要求 |
| --- | --- |
| JDK | 21 或以上 |
| MySQL | 8.0 或以上 |
| Redis | 7.x，演示 Redis 榜单时需要 |
| Maven | 可使用项目自带 `mvnw` |
| Node.js | 前端开发和构建时需要 |

### B.2 初始化数据库

新库或演示库重置时，执行：

```bash
mysql -u root -p --default-character-set=utf8mb4 < src/main/resources/sql/00_setup_fresh_database.sql
```

已有旧库升级时，执行：

```bash
mysql -u root -p --default-character-set=utf8mb4 < src/main/resources/sql/01_upgrade_existing_database.sql
```

详细执行顺序见：

```text
src/main/resources/sql/数据库脚本执行顺序说明.md
```

### B.3 启动后端

```bash
./mvnw spring-boot:run
```

Windows PowerShell 也可以使用：

```powershell
.\mvnw.cmd spring-boot:run
```

默认数据库配置位于：

```text
src/main/resources/application.yml
```

### B.4 启动或构建前端

前端开发模式：

```bash
cd frontend
npm install
npm run dev
```

构建前端并交给 Spring Boot 静态资源访问：

```bash
cd frontend
npm install
npm run build
```

### B.5 Redis

Redis 默认连接：

```text
localhost:6379
```

在 WSL 或 Linux 中可以使用：

```bash
sudo service redis-server start
redis-cli ping
```

返回 `PONG` 表示 Redis 正常。
